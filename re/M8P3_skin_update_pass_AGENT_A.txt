================================================================================
  EXECUTIVE SUMMARY (top of mind)
================================================================================

  We hunted the function that, per the GPT analysis we were given:
      "reads each bone's world matrix, multiplies by inverse-bind, and writes
       the result into a flat per-bone matrix buffer that is then uploaded
       to the GPU as a constant buffer."

  PRIMARY FINDING: the GPT description is INCORRECT for this engine, OR it
  describes a function that is split across two layers and isn't visible as a
  single CPU routine. Specifically:

    (a) BSSkin::Instance vtable (RVA 0x267E5C8) has NO skin-update virtual.
        All 40 slots are ctor/dtor/serialize/clone — see Section 1.

    (b) The CPU never performs `world * inv_bind` for the skin update pass.
        The only function that does that math (sub_1403FA7C0, RVA 0x3FA7C0)
        is a BONE-BY-NAME RESOLVER for animation graph queries, NOT a
        per-frame walker. See Section 4.

    (c) NiSkinInstance vt slot 28 (sub_1416EEBE0, RVA 0x16EEBE0) builds a
        cached flat array of POINTERS-TO-BONE-WORLD-MATRICES at NSI+0x30.
        Each entry = bone_AVObject + 0x70 (the start of the world matrix).
        See Section 2 for full annotated decomp.
        This array is built ONCE at NIF deserialization (vt slot 28 = Load),
        not per-frame, but the pointers stay valid as long as bones exist.
        Per-frame consumer of this cache is TBD-1 below.

    (d) Plausible mechanism: at render time, the engine SETS the skinning
        constant buffer parameters by walking NSI+0x30 and writing each
        bone's matrix (read via the cached pointer) into a flat upload
        buffer. The inverse-bind matrices are uploaded SEPARATELY as a
        per-geometry static cb (bound once when the geom is registered).
        The shader does `world * inv_bind` per vertex.

  TOP CANDIDATE for the per-frame uploader is unconfirmed — it's downstream
  of NSI but not catchable via vtable scan or string xref. Two plausible
  RVA-traced candidates:
      sub_140C5F0E0 (RVA 0xC5F0E0, size 0xE08) — UpdateSubresource caller,
                                                  references skin-related
                                                  offsets, render-time fn
      sub_1404D2240 (RVA 0x4D2240, size 0xB32) — Map (vt+0x70) caller,
                                                  large per-frame engine fn

  Confidence: 35% — the structural map of NiSkinInstance is solid; the
  per-frame uploader hypothesis needs additional empirical validation
  (Frida tracing or Agent B re-investigation).

================================================================================
  SECTION 2 — Strategy 2: NiSkinInstance LOAD path = the cache builder
================================================================================

  *** TOP STRUCTURAL FINDING ***

  NiSkinInstance is a PARALLEL hierarchy to BSSkin::Instance — DIFFERENT class,
  DIFFERENT vt (RVA 0x2680DA8). It exists in the binary but BSSkin::Instance
  is what BSGeometry uses (per M8P3 dossier confirmed via geom+0x140 read).

  HOWEVER, NiSkinInstance vt slot 28 (`sub_1416EEBE0`, RVA 0x16EEBE0, sz 0x13A)
  has the EXACT signature we'd want for the skin update pass — except it's
  the LOAD function, not the per-frame updater.

  Annotated decomp:

      __int64 __fastcall sub_1416EEBE0(NiSkinInstance* a1, NiStream* a2)
      {
        // a1[2] = NiSkinData ptr        (NSI+0x10)
        // a1[3] = NiSkinPartition ptr   (NSI+0x18)
        // a1[4] = NiAVObject ptr        (NSI+0x20)  — skel root
        // a1[5] = bones array head      (NSI+0x28)  — raw NiAVObject** (8B/entry)
        // a1[6] = cached pointer array  (NSI+0x30)  — array of bone+0x70 pointers

        nullsub_587();

        // Read skinData ref + skinPartition ref + skel root ref from stream
        a1[2] = sub_1416DE030(a2);   // NiSkinData
        a1[3] = sub_1416DE030(a2);   // NiSkinPartition
        a1[4] = sub_1416DE030(a2);   // skel root NiAVObject

        // Read bone count from stream
        v8 = sub_1416DE090(a2);      // u32 bone_count
        a1[5] = sub_141677A80(8 * v8);  // alloc bones[]
        for (i=0; i<v8; ++i)
            a1[5][i] = sub_1416DE030(a2);  // read each bone NiPointer

        // Build cached pointer array: for each bone, store its world-matrix start
        v11 = *(uint32_t*)(a1[2] + 0x58);   // bone count from NiSkinData+0x58
        a1[6] = sub_141677A80(8 * v11);     // alloc cache (NSI+0x30)
        for (i=0; i<v11; ++i) {
            v14 = a1[5][i];                 // NiAVObject* bone
            v15 = v14 + 112;                // bone+0x70 = world matrix start
            if (!v14) v15 = 0;
            a1[6][i] = v15;                 // store ptr to world-matrix slot
        }
        return a1[6];
      }

  KEY OBSERVATION: the cached array a1[6] (NSI+0x30) contains POINTERS to each
  bone's world matrix at +0x70. To upload skin matrices to GPU, the engine
  just needs to:
      for (i=0; i<count; ++i) {
          memcpy(out + i*64, *cached[i], 64);  // copy 4x4 from bone.world
      }
  This is what we'd expect to find as the "flat buffer build" function.

  Or alternatively, the GPU buffer is filled directly via cached[i]->m128
  loads with no intermediate.

  CALLERS of sub_1416EEBE0: 0 direct callers (it's an indirect dispatch from
                            the NIF stream loader walking class vt slot 28).

  HOOK FEASIBILITY: hooking sub_1416EEBE0 lets us LEARN when a NiSkinInstance
  was just deserialized; useful for marking ghost skins.

  SECTION 4 — Strategy 3b: per-bone matrix multiply found, but for ANIMATION
================================================================================

  We did find a function that multiplies bone.world × inv_bind: sub_1403FA7C0
  (RVA 0x3FA7C0, sz 0x1B4). Annotated decomp:

      __int64 sub_1403FA7C0(__int64 a1, __int64 a2)
      {
        // a1 = "names container" (BSTArray<NameRecord>)
        //   a1[0]    = head ptr
        //   a1[16]   = count
        // a2 = context with:
        //   a2[0]  = ptr to skin instance ptr
        //   a2[8]  = ptr to bone-name BSFixedString
        //   a2[16] = ptr to OUTPUT 4-xmm matrix
        //   a2[24] = ptr to bone-index to update

        v2 = a1[16];                    // count
        v4 = a1[0];                     // head
        while (1) {
            v6 = *v4;                   // entry ptr
            if (sub_1403FA450(skin, v6+16, v22, name)) break;  // matcher
            ++v4;
            ++idx;
        }

        // After match, read inv_bind from boneData (stride 0x50, +0x10):
        v8  = (uint*)(a2+24);
        v10 = *(skin->boneData) + 16;     // boneArray head
        v20[0] = (m128*)(v10 + 0x50*idx + 0x10);   // row 0 of inv_bind
        v20[1] = (m128*)(v10 + 0x50*idx + 0x20);   // row 1
        v20[2] = (m128*)(v10 + 0x50*idx + 0x30);   // row 2
        v21    = (m128*)(v10 + 0x50*idx + 0x40);   // row 3

        // Multiply bone.world (v22) × inv_bind (v20)
        v12 = sub_1403444F0(v22, v24, v20);

        // Write result into output (4 xmm rows at v14[6..9]):
        v14[6] = v12[0]   // row 0
        v14[7] = v12[1]   // row 1
        v14[8] = v12[2]   // row 2
        v14[9] = v12[3]   // row 3
      }

  CALLERS: ONE — sub_1403F7AD0 (RVA 0x3F7AD0, sz 0xDC) which has 0 callers
  (likely indirect dispatch). Sister fn sub_1403FA980 has 2 callers
  sub_1403F74F0 + sub_1403FABF0.

  PURPOSE: These are AnimationGraph "GetBoneTransform(name) -> matrix" queries,
  used during animation evaluation to find a bone's world transform for IK,
  effects placement, etc. They match per-bone but iterate by NAME and
  multiply EACH on demand. Not a per-frame walker over all bones.

  RVAs of related fns:
      sub_1403FA7C0   per-bone resolver A (sister A)
      sub_1403FA980   per-bone resolver B (sister B)
      sub_1403F7AD0   caller of FA7C0 (single)
      sub_1403F74F0   caller of FA980 (one of two)
      sub_1403FABF0   caller of FA980 (recursive — walks tree)
      sub_1403FA450   matcher (compares bone name to records)

      __int64 sub_1421B4730(BSGeometry* geom, RenderContext* ctx)
      {
        // Read skin instance from BSGeometry+0x140
        skin = *(BSSkin::Instance**)((char*)geom + 0x140);

================================================================================
  SECTION 10 — How it computes per-bone matrix
================================================================================

  PER OUR ANALYSIS: it does NOT. The CPU just COPIES bone+0x70 (the world
  matrix) into a flat upload buffer. The shader does the multiply by
  inv_bind during vertex skinning.

  Evidence:
    - `world * inv_bind` only appears in 1 fn family (sub_1403FA7C0, etc.)
      and that's an animation-graph query, not a render walker.
    - The boneData->boneArray (stride 0x50) is read ONLY by destructors —
      never by render-time code (verified Section 3 above).

  This contradicts the GPT analysis we were given — see TBD-2 / Confidence.

================================================================================
  SECTION 11 — Where does it write the output buffer?
================================================================================

  PRESUMED behavior:
    - The output buffer is a D3D11 dynamic constant buffer (not heap-alloc,
      not stack-alloc). It is owned by BSGraphics::Renderer (qword_1434380A8).
    - Map() returns a CPU-writable pointer; the engine writes 4-xmm rows
      per bone into consecutive 64-byte slots; Unmap() commits.

  This pattern is standard D3D11 dynamic cb upload.

  Buffer size: 58 bones × 64 = 3712 bytes for player skeleton, OR
              up to 64 (vtx skin shader limit) × 64 = 4096 bytes max.

================================================================================
  SECTION 12 — How is it uploaded to GPU
================================================================================

  Standard D3D11 path:
      1. ctx->Map(cb, 0, D3D11_MAP_WRITE_DISCARD, 0, &subresource)
      2. memcpy(subresource.pData, srcMatrices, count*64)
      3. ctx->Unmap(cb, 0)
      4. ctx->VSSetConstantBuffers(register_slot, 1, &cb)

  Once bound, the vertex shader reads bone matrices and inv_bind from CBs and
  computes the final transformed vertex. No CPU multiply involved.
